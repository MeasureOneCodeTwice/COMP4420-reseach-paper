Hardware advancements have scaled unevenly over the decades, with processor clock speeds advancing much faster than memory fetching speeds~\cite{wulf_memorywall}. This problem is exacerbated when data sizes grow into terabyte, petabyte, or lately, exabyte levels~\cite{boureima_exabyte}. Distributed architectures are increasingly common, where storage is separated from computation, including the complete disaggregation of memory~\cite{shan_disaggregation}. While the original problem can be motivated by the hardware and data on a home computer, it becomes exponentially costlier when external networks become the data bus with network latency.

Even at a small scale, consider a modern home computer with components purchased within the last year, equipped with Viper Xtreme 5 DDR5 6000MT/s CL30 memory and an AMD Ryzen 5 9600X 5.4GHz processor. According to standard latency calculations, fetching a block of memory for the first time requires 10 ns: 
\[
    \text{Latency (ns)} = \frac{2000 \cdot \text{CAS Latency}}{\text{RAM speed in MT/s}} = \frac{2000 \cdot 30}{6000} = 10 \text{ ns} 
\]
Within that same 10 ns time frame, the processor has already completed 54 clock cycles, as derived from its clock speed of 5.4 GHz. Without advanced hardware prefetchers, this would mean that the processor stalls for at least 54 clock cycles while waiting for data. And this is worse when you consider that all you likely wanted was a single word in the fetched block, underutilizing all the words that came alongside in that same block.

This reveals a problem with the Word RAM model of algorithm analysis. It considers all memory fetches to be completely free, assigning costs only to processor instructions. Clearly this is a na\"{\i}ve view of the real world time complexity of this algorithm, as an algorithm that thrashes the memory would underperform one of similar Word RAM model runtime complexity that consumes all words in every block it fetches.

In this paper, we will investigate cache optimization through the lens of the Cache-Oblivious model. Then we will explore a few algorithms, namely \textit{lazy funnelsort}, and examine what makes them cache optimal on all hardware. We will focus the algorithms section primarily on the lazy funnelsort algorithm, including our attempt at implementing it. While our original proposal targeted benchmarking this algorithm alongside other cache-oblivious algorithms, the complexity of designing and implementing optimal cache-oblivious algorithms proved challenging and time-consuming. Therefore, we will discuss the challenges we faced during the design and implementation of lazy funnelsort, the gap between the theoretical funnelsort model and real-world programming, and potential for future work built upon our groundwork.

\section{The Cache-Oblivious Model}
% 1.5 - 2 pages
% - 0.5pg External Memory Model
% - 1pg Cache-Oblivious Model

Since cache optimality is our goal, we have already seen why the Word RAM model of analysis is not the optimal lens by which to analyze this. Though we have already alluded to us wanting to be oblivious to hardware specifications with our analysis, forcing the algorithms to be cache optimal regardless of the hardware it is running on, we believe that the cache-oblivious model is best understood intuitively by building up from analysis models that \textit{do} know hardware specifications. Now that we have our basic memory layout defined, we can start with the External Memory Model.

The External Memory Model observes only two arbitrary levels within the cache hierarchy~\cite{demaine}. The higher level of these two levels will be our cache, and the lower level will be our memory~\ref{fig:cache-visual}. Our cache has a total size of $M$ words, and we assume the memory contains $> M$ words~\cite{demaine}. Data is stored and transferred in chunks, called blocks. Each block consists of $B$ words~\cite{demaine}. Thus, our cache has $M/B$ cache lines, which is where blocks are stored when fetched from memory~\cite{demaine}. 

% External Memory Model: program must decide which cache line to evict
Every time a program needs data for the first time, it will need to load that from memory. Since it cannot simply fetch that word by itself, the entire block of $B$ words, one of which is the word the program requested, is transferred. If all cache lines are occupied, then the program must also first decide which cache line to empty by evicting the block from this cache line~\cite{demaine}. Since the program is only interested in one word, it wastes the expensive memory transfer of all $B-1$ words that were unutilized in this block.

The cost of transferring this block across the bus, and the associated latency of requesting it from memory, would be better amortized if we were able to use all $B$ words within this block. And we would like to do that without even knowing the hardware specific $B$ and $M$ parameters.

% Ideal cache: machine where programs can run either cache-aware, or cache-oblivious
So far under the External Memory Model, the program is responsible for the optimal cache replacement strategy~\cite{demaine}. We can abstract this responsibility away and remove knowledge of hardware parameters by introducing the Cache-Oblivious Model~\cite{demaine}. 

Similar to the External Memory Model, the Cache-Oblivious Model is a theoretical machine that consists only of two levels of cache hierarchy, the small, fast cache and the slow, large memory~\cite{demaine}.\footnote{Cache obliviousness was originally introduced as a chosen property of the Ideal-Cache Model by Frigo et al., where the model was still aware of hardware parameters but the algorithms could choose to ignore them, thus making them cache oblivious~\cite{frigo}. We will follow Demaine's lead in introducing the Cache-Oblivious Model to hold the cache oblivious properties built atop the Ideal-Cache Model~\cite{demaine}.} 

Contrary to the External Memory Model, the Cache-Oblivious Model places the responsibility of the optimal cache replacement strategy on the machine itself and not the programs run on it~\cite{demaine}. This optimal cache replacement strategy is one where the cache line holding the block that will be used furthest in the future is the one that is evicted next, ensuring temporal locality~\cite{demaine}. Demaine assures us that this strategy can be substituted, without loss of optimality, with the Least Recently Used (LRU) eviction strategy~\cite{demaine}. 

The Cache-Oblivious Model also takes on the Tall Cache assumption, $M \in \Omega(B^2)$, which assumes that the cache itself contains $\ge B^2$ words~\cite{demaine}. This will ensure that the number of cache lines $= M/B \ge B^2/B = B$, meaning that it is at least as tall as it is wide. 

Finally, the Cache-Oblivious Model also assumes that the cache is fully associative, which is $M/B$-way associative, which ensures that any block in memory can be allocated to any cache line~\cite{demaine}. 

Frigo et al. and Demaine observe that a beneficial side-effect of algorithms being cache oblivious is that they are optimal across all levels of cache, not just the two levels within our theoretical model~\cite{frigo,demaine}. The reason this property unfolds is that we can examine \textit{any} two levels of the cache hierarchy with our Cache-Oblivious Model. Since the algorithm within the Cache-Oblivious Model is cache optimal without any knowledge of hardware parameters, namely the block size $B$ or cache size $M$, then the algorithm must be cache optimal across any two adjacent levels of the cache hierarchy. This side effect holds as long as the cache hierarchy maintains the \textit{inclusion property}, meaning that any block stored in a higher, faster cache level is also stored in the lower, slower memory level~\cite{frigo,demaine}.

With the Cache-Oblivious Model in hand, we can now build programs on top of it without ever relying on hardware parameters, $B$ and $M$, to make them cache optimal. Thus, we have the foundation by which we will study \textit{lazy funnelsort}.

Before examining \textit{lazy funnelsort}, we must first understand the underlying memory layout that enables its cache optimality: the cache-oblivious van Emde Boas trees.

\section{Cache-Oblivious van Emde Boas Trees}
% 1 - 1.5pg

To lay the groundwork for the data structure that enables a cache optimal \textit{lazy funnelsort} algorithm, we need to examine cache-oblivious van Emde Boas (vEB) trees. We assume the reader is already familiar with standard vEB trees, as are usually studied within the Word RAM model. We will adapt the vEB data structure to the Cache-Oblivious Model to optimally store large datasets. Accordingly, we will focus only on the static vEB tree, exploring only its memory layout and space complexity while ignoring dynamic operations.

A cache-oblivious vEB tree of $N$ nodes is recursively divided into a \textit{summary} vEB tree of size $\sqrt{N}$ and $\sqrt{N}$ \textit{cluster} vEB trees, each also of size $\sqrt{N}$~\cite{demaine}. Conceptually, the tree's height is divided in half, with the summary tree forming the top half and the clusters forming the bottom half~\cite{demaine} (Figure~\ref{fig:veb-tree}). To optimally store this data, these subtrees are laid out sequentially within a single contiguous segment of memory: the summary tree is placed first, followed immediately by the cluster trees ordered from left to right~\cite{demaine} (Figure~\ref{fig:veb-layout}).

Since the vEB tree recurses into smaller subtrees stored sequentially in memory, the recursion guarantees that at some level, the subtrees will reach a size of $\le B$ words~\cite{demaine}. This subtree would therefore occupy at most two consecutive memory blocks as it may not be perfectly block aligned, and thus require at most two memory transfers to load the entire subtree into cache~\cite{demaine}. Any subsequent memory operations within this subtree would require no additional memory transfers, having an $O(1)$ cache complexity. 

We are now well equipped to study the \textit{lazy funnelsort} algorithm. The vEB memory layout will be key in making the \textit{lazy funnelsort} cache optimal.

\section{Lazy Funnelsort}

Funnelsort may be a novel sorting algorithm, but its roots are in mergesort. Let's build up to the funnelsort algorithm through targetting cache optimality in mergesort.

Firstly, lets analyse in depth the most basic form of merge sort, a two way merge sort. For an input array of size $n$ this algorithm recursively splits the input array into $2$ halves of size $h$. These halves are sorted (by merge sort) and merged together by performing a linear scan on each input array and writing in a linear scan to an output array. Given the array is not already in cache, scanning each input array costs $\frac{h}{B}$ memory transfers and writing to the output array costs $\frac{2h}{B}$ memory transfers. From this we create the cost tree in Figure \ref{fig:two-way-merge-sort-cost-tree}. The cost of each layer of the tree sums to $4\frac{n}{B}$ and the tree has height $log_2n$.

    \\ & 
    \Rightarrow 
    B \geq 
    (\log_2n - d)\frac{n}{2^{d}}

    \Rightarrow 
    B \geq

\subsection{How it works}
Similar to merge sort, funnel sort recursively splits the input into equal sized chunks and merges them together.
The K-Funnel data structure is how funnel sort simulates a $\frac{M}{B}$ way merger and is the bulk of the of the complexity of the algorithm.
Each K-Funnel takes $K$ sorted input buffers of size $K^2$ and merges them into to a single buffer of size $K^3$.
The K-Funnel's recursive structure is similar to that of a Emde Boas tree.

A K-Funnel is comprised of $\sqrt{k} + 1$ $\sqrt{k}$ funnels.
It has 1 top funnel and $\sqrt{k}$ bottom funnels.
Each bottom funnel merges $\sqrt{k}$ of the K-Funnel's input buffers and writes them to a size $(\sqrt{k})^3$ output buffer.
These internal buffers, of which there are $\sqrt{k}$ are used as inputs to the top $\sqrt{k}$ funnel which merges the their contents into the K-Funnel's output buffer.
